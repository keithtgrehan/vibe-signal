# Monitoring And No-Raw-Log Readiness

Status: closed-beta monitoring scaffold only. This document does not claim production monitoring, production compliance, legal compliance, GDPR/CCPA readiness, or public-launch readiness.

## Implemented In This Repo

- `GET /healthz` provides a minimal liveness check.
- `GET /readyz` provides readiness metadata without echoing configured origins or secret values.
- Backend requests emit metadata-only operational log events through the standard Python logger.
- Backend responses include a server-generated `X-Request-ID` so an operator can correlate client reports with metadata logs.
- Unexpected backend exceptions return a generic error with a request ID instead of exposing stack traces or raw exception text to users.
- Safe request logging reports:
  - request ID
  - HTTP method
  - safe route template or endpoint category, not arbitrary request path text
  - status code
  - status category
  - latency bucket
  - coarse error category
  - explicit false flags for raw body, raw message, provider response, and secret logging

## Not Implemented

- No production monitoring provider is configured.
- No uptime SLA is claimed.
- No alerting, paging, dashboard, or incident automation is implemented.
- No production incident-response process is complete.
- No server, proxy, or platform access-log configuration is committed; those logs must be disabled or sanitized in the deployment provider before beta traffic.
- No analytics or tracking SDK is added.
- No request body, raw chat, provider response, secret, model artifact, vector, checkpoint, or cached embedding storage is added.
- No model-quality, legal-compliance, GDPR/CCPA, or production-readiness claim is made.

## No-Raw-Data Logging Policy

Never log:

- raw user messages
- request bodies for `/api/analyze` or `/api/match`
- third-party private messages
- provider responses
- API keys, bearer tokens, passwords, secrets, or credentials
- full support messages
- arbitrary URL paths, query strings, authorization headers, or cookies from server/proxy access logs
- model vectors, embeddings, checkpoints, cached artifacts, or generated model files
- raw external dataset rows

Allowed operational metadata:

- request ID
- timestamp generated by the deployment log system
- safe route template or endpoint category without query string
- HTTP method
- status code
- status category
- latency bucket
- coarse error category
- deployment environment label
- safe readiness metadata from `/readyz`

## Closed-Beta Monitoring Checks

Run these checks against the deployed backend host after each closed-beta deploy. Use a server command or provider setting that disables/sanitizes default access logs, such as `uvicorn ... --no-access-log`, so the safe metadata logger is the operational request log:

```bash
curl https://<your-backend-host>/healthz
curl https://<your-backend-host>/readyz
curl https://<your-backend-host>/legal/match-disclaimer
```

Run one synthetic match smoke test. Do not use private chats:

```bash
curl -sS https://<your-backend-host>/api/match \
  -H 'Content-Type: application/json' \
  -d '{
    "conversation_id": "synthetic_closed_beta_smoke",
    "messages": [
      {"id": "m1", "author": "self", "text": "Can you confirm Friday at 3pm?"},
      {"id": "m2", "author": "other", "text": "Yes, Friday at 3pm works. No pressure if we need to adjust."}
    ]
  }'
```

If event routes are in scope for that deploy:

```bash
cd mobile
npm run verify:backend -- --api-url https://<your-backend-host> --event state
```

Manual log review:

- Confirm smoke-test responses and reviewed error responses have server-generated request IDs; investigate any safe backend log event without one.
- Confirm safe endpoint names/categories, status codes, and latency buckets are visible.
- Confirm query strings are not logged.
- Confirm server/proxy/platform access logs are disabled or sanitized.
- Confirm raw message text, request bodies, provider responses, credentials, tokens, vectors, and checkpoints do not appear.
- Record the date, backend host, git SHA, environment label, and reviewer initials in the release notes or deployment tracker.

## Suggested Alert Thresholds

These are candidate closed-beta operator thresholds, not an SLA. Once a monitoring provider exposes rolling windows, use these thresholds for alerts; before that, review sampled metadata logs manually after deploys and after user-reported errors:

- uptime check fails twice in a row
- `/readyz` returns `not_ready`
- `/readyz` reports unsafe logging or persistence flags
- 5xx responses exceed 2% over 15 minutes
- p95 latency is above 2 seconds over 15 minutes
- any raw message, request body, provider response, or secret appears in logs
- any unsafe user-facing wording appears in backend responses

## Incident Response Runbook

1. Freeze promotion of the current deploy.
2. Capture the request ID, route, timestamp, status code, latency bucket, and coarse error category.
3. Do not copy raw user messages into incident notes.
4. Check `/healthz` and `/readyz`.
5. Review recent metadata-only logs for route-level patterns.
6. If raw content appears in logs, treat it as a launch blocker, restrict access, pause traffic if needed, preserve/delete according to provider retention and legal review, and document exposure scope.
7. Roll back or disable the affected route when the issue is reproducible, privacy-sensitive, or user-visible.
8. Document the issue, fix, validation, and remaining risk before resuming beta traffic.

## Rollback Triggers

Roll back or pause beta traffic when any of these occur:

- raw user content appears in logs
- secrets, provider responses, vectors, checkpoints, or model files appear in logs or committed artifacts
- `/api/match` emits hidden-intent, attraction, cheating, diagnosis, neurotype, attachment-style, emotional-truth, or manipulation claims
- `/readyz` returns `not_ready` for hard safety flags
- repeated 5xx responses affect the match or analysis routes
- deletion/export or privacy copy is contradicted by deployed behavior

## Remaining Launch Blockers

- Live deployment monitoring provider and dashboard selection.
- Alert routing and incident owner assignment.
- Final reviewed privacy, terms, deletion, and export URLs.
- Legal-reviewed deletion/export support flow and retention policy.
- Real reviewed labels before public model-quality claims.
- Commercial-safe training rights before any commercial training use.
